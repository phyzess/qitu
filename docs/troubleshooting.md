# Troubleshooting

Status: draft  
Date: 2026-06-27

## `vp install` Cannot Reach `registry.npmjs.org`

Typical error:

```text
error sending request for url (https://registry.npmjs.org/pnpm/-/pnpm-11.5.2.tgz)
tcp connect error: Connection refused
```

Meaning:

The project is registry-neutral, but the current machine or network cannot reach the public npm registry.

Check local environment without printing secrets:

```sh
vp run doctor
```

Things to inspect:

1. Company proxy or VPN rules.
2. User-level npm registry config.
3. `NPM_CONFIG_REGISTRY` environment variable.
4. Firewall rules blocking `registry.npmjs.org`.

Do not commit an internal registry into `.npmrc` for this reusable starter. If a specific company needs one, keep it in user-level config or a private downstream fork.

## TypeScript 7 Platform Package Fails

`typescript@7.0.2` installs the official stable TypeScript 7 package and may resolve a platform package such as:

```text
@typescript/typescript-darwin-arm64@7.0.2
```

If that package fails to download, the problem is usually registry reachability or mirror freshness, not the project manifest.

Recommended response:

1. Keep `typescript` pinned to `7.0.2`.
2. Confirm the registry can serve the platform package.
3. Avoid replacing it with `typescript@6` unless a new decision is recorded.

## `pnpm peers check` Throws an Internal Error

Observed with `pnpm@11.5.2`:

```text
Cannot read properties of undefined (reading 'devDependencies')
```

This is not a useful peer dependency report. Treat it as a pnpm command failure until proven otherwise.

Current workaround:

1. Keep `pnpm-lock.yaml` generated by `pnpm install --lockfile-only --ignore-scripts`.
2. Use `vp run smoke` for template invariant checks.
3. Use `vp run validate` once dependencies are installed and Vite+ checks can run.
4. Revisit `pnpm peers check` after upgrading pnpm through an explicit dependency decision.

## Wrangler Type Generation Fails

Run:

```sh
vp run cf:typegen
```

If it fails:

1. Confirm dependencies are installed.
2. Confirm `apps/worker/wrangler.jsonc` is valid JSONC.
3. Confirm `wrangler@4.103.0` is available.
4. Delete stale `apps/worker/worker-configuration.d.ts` only if you intend to regenerate it immediately.

## Local D1 Migration Fails

Run:

```sh
vp run db:migrate:local
```

If it fails:

1. Confirm `apps/worker/migrations` exists.
2. Confirm the D1 database name in `apps/worker/wrangler.jsonc` is `qitu-dev`.
3. Confirm Wrangler can create local state under `.wrangler/`.

## Import Job Stays Queued

Expected baseline:

```text
queued -> processing -> needs_review
queued -> processing -> failed
```

If jobs remain `queued`:

1. Confirm the Queue consumer is running through `vp run dev` or `vp run dev:worker`.
2. Confirm `IMPORT_JOBS` is bound in `apps/worker/wrangler.jsonc`.
3. Query local jobs:

```sh
npx wrangler d1 execute qitu-dev --local --config apps/worker/wrangler.jsonc --command "SELECT id,status,failure_reason FROM import_jobs ORDER BY created_at DESC LIMIT 5"
```

## R2 Object Exists Without Metadata

Upload writes R2 first, then D1 metadata. If the D1 write fails, the Worker attempts to delete the R2 object.

If an orphan remains:

1. Check `source_files.object_key`.
2. Check the R2 object key under `SOURCE_FILES`.
3. Delete orphaned local test data or rerun from a clean local state.

Future recovery work should add an explicit reconciler before production use.

## Queue Dispatch Fails After Upload

If Queue dispatch fails after source metadata is written, the Worker marks the import job `failed` and records a failure reason.

Query:

```sh
npx wrangler d1 execute qitu-dev --local --config apps/worker/wrangler.jsonc --command "SELECT id,status,failure_reason FROM import_jobs WHERE status = 'failed' ORDER BY updated_at DESC LIMIT 5"
```

Retry a failed job through the app UI or API after the underlying cause is fixed:

```sh
curl -X POST http://127.0.0.1:8787/api/import-jobs/<job-id>/retry
```

The retry route requires an authenticated session. It resets the job to `queued`, clears the stored failure fields, sends a new Queue message, and writes an `import_job.retry_queued` audit event.

## `dev` Starts One Process Then Exits

`vp run dev` starts both web and Worker dev servers. If one process exits with a non-zero code, the wrapper stops the other process.

Run each side separately to isolate the failure:

```sh
vp run dev:web
vp run dev:worker
```

## Generated Files Look Stale

The starter ignores generated build output with `.gitignore`.

If `apps/web/dist` appears during local work, it is build output. Do not treat text inside it as source truth. Change `apps/web/src/*` and rebuild.

## AI Output Appears in Committed Data

This violates the default policy.

Rules:

1. AI may suggest, classify, extract, or explain.
2. AI output must be stored as advisory records.
3. A human operator must confirm before business-owned records are committed.
4. Audit events must link the confirmer, source file, advisory record, and commit.
